CHAPTER 01

Orientation and Execution Model

Interpreted vs. compiled workflows, the Python runtime, virtual environments, and development setup.

Learning Outcomes

  • ●Contrast Python's interpreted execution model with the compiled model of C++ and the JVM-based model of Java.
  • ●Run Python code interactively and as standalone scripts from the command line.
  • ●Create, activate, and manage virtual environments for dependency isolation.
  • ●Install third-party packages using pip and understand requirement files.
  • ●Identify the main components of a Python development workflow.

1.1 Why Python?

Python is a high-level, general-purpose programming language that emphasizes readability, conciseness, and developer productivity. It was created by Guido van Rossum and first released in 1991. Today it is among the most widely used languages in industry, powering applications in web development, data science, automation, embedded systems, and scientific computing.

For students with a C++ and Java background, Python represents a shift in design philosophy. Where C++ prioritizes low-level control and performance, and Java emphasizes type safety and portability through the JVM, Python optimizes for rapid iteration and expressive syntax. The trade-off is reduced raw execution speed, which is acceptable for most engineering workflows because the bottleneck is typically I/O, not computation.

1.2 Interpreted vs. Compiled Execution

In C++, source code is compiled to native machine instructions by a compiler such as g++ or clang++. In Java, source is compiled to bytecode and then executed by the Java Virtual Machine (JVM). Python takes a different approach: the CPython interpreter translates source code to bytecode at runtime and immediately executes it on a virtual machine embedded in the interpreter itself.

C++ Workflow

Source → Compiler → Machine code → Execute

Java Workflow

Source → javac → Bytecode (.class) → JVM

Python (CPython) Workflow

Source → Interpreter → Bytecode (in memory / .pyc) → CPython VM

Key Consequence

No separate compilation step. Errors are detected at runtime, not at compile time.

Because Python is dynamically typed and interpreted, type errors that C++ or Java would catch at compile time only surface when the offending line is executed. This makes thorough testing especially important.

1.3 Running Python Code

Python code can be executed in two modes: interactive (REPL) and script-based.

Interactive Mode (REPL)

The Read-Eval-Print Loop is launched by typing python in a terminal. Each expression is evaluated immediately, making it ideal for exploration and debugging.

>>> 2 + 3
5
>>> print("Hello from the REPL")
Hello from the REPL

Script Mode

A Python file (conventionally with a .py extension) is executed by passing it to the interpreter.

# main.py
import sys

print(f"Python version: {sys.version}")
print("Script execution complete.")

The if __name__ == "__main__" Guard

When a module is run directly, Python sets its __name__ attribute to "__main__". This guard prevents code from executing when the file is imported as a module.

def greet(name):
    return f"Hello, {name}"

if __name__ == "__main__":
    print(greet("Engineer"))

1.4 Virtual Environments and Package Management

A virtual environment is an isolated directory tree that contains a specific Python interpreter and a set of installed packages. This prevents version conflicts between projects.

# Create a virtual environment
python -m venv myenv

# Activate (Windows)
myenv\Scripts\activate

# Activate (Linux / macOS)
source myenv/bin/activate

# Install packages
pip install requests numpy

# Freeze current dependencies
pip freeze > requirements.txt

# Reproduce environment elsewhere
pip install -r requirements.txt
Always work inside a virtual environment to keep your system Python installation clean and to ensure reproducibility of your projects.

1.5 Development Workflow Overview

A typical Python development cycle involves the following steps:

  1. Explore — Use the REPL to test small ideas and examine APIs.
  2. Write — Organize code into modules and packages within a project directory.
  3. Test — Run unit tests with pytest or unittest after each change.
  4. Debug — Use print(), the built-in pdb debugger, or IDE breakpoints.
  5. Ship — Package the project and manage dependencies with pip and pyproject.toml.

This tight feedback loop is one of Python's greatest strengths. Unlike C++ or Java, there is no waiting for a compilation step, which significantly accelerates prototyping and iterative development.
