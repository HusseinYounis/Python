CHAPTER 01

Orientation and Execution Model

Interpreted vs. compiled workflows, the Python runtime, virtual environments, and development setup.

Learning Outcomes

  • ●Contrast Python's interpreted execution model with the compiled model of C++ and the JVM-based model of Java.
  • ●Run Python code interactively and as standalone scripts from the command line.
  • ●Create, activate, and manage virtual environments for dependency isolation.
  • ●Install Python using the official installer, Anaconda, or Miniconda and understand the differences.
  • ●Install third-party packages using pip and conda and understand requirement files.
  • ●Identify the main components of a Python development workflow.

1.1 Why Python?

Python is a high-level, general-purpose programming language that emphasizes readability, conciseness, and developer productivity. It was created by Guido van Rossum and first released in 1991. Today it is among the most widely used languages in industry, powering applications in web development, data science, automation, embedded systems, and scientific computing.

For students with a C++ and Java background, Python represents a shift in design philosophy. Where C++ prioritizes low-level control and performance, and Java emphasizes type safety and portability through the JVM, Python optimizes for rapid iteration and expressive syntax. The trade-off is reduced raw execution speed, which is acceptable for most engineering workflows because the bottleneck is typically I/O, not computation.

1.2 Interpreted vs. Compiled Execution

In C++, source code is compiled to native machine instructions by a compiler such as g++ or clang++. In Java, source is compiled to bytecode and then executed by the Java Virtual Machine (JVM). Python takes a different approach: the CPython interpreter translates source code to bytecode at runtime and immediately executes it on a virtual machine embedded in the interpreter itself.

C++ Workflow

Source → Compiler → Machine code → Execute

Java Workflow

Source → javac → Bytecode (.class) → JVM

Python (CPython) Workflow

Source → Interpreter → Bytecode (in memory / .pyc) → CPython VM

Key Consequence

No separate compilation step. Errors are detected at runtime, not at compile time.

Because Python is dynamically typed and interpreted, type errors that C++ or Java would catch at compile time only surface when the offending line is executed. This makes thorough testing especially important.

1.3 Setting Up Your Python Environment

Before writing any Python code you need a working Python installation. There are three common approaches: the official CPython installer, Anaconda, and Miniconda. Each has different trade-offs in size, convenience, and target audience.

Option 1 — Official Python Installer (python.org)

The simplest way to get Python is to download the installer from python.org. This gives you the standard CPython interpreter and pip for package management.

# Step 1 - Download the installer from https://www.python.org/downloads/
# Step 2 - Run the installer
#   IMPORTANT: Check "Add python.exe to PATH" during installation (Windows)
# Step 3 - Verify
python --version
# Python 3.12.x

pip --version
# pip 24.x from ...

This option is lightweight and gives you a clean, minimal setup. You manage virtual environments with the built-in venv module and install packages with pip.

Option 2 — Anaconda

Anaconda is a full-featured data science distribution. It ships with Python, conda (its own package and environment manager), and over 250 pre-installed scientific packages including NumPy, Pandas, Matplotlib, Jupyter, and Scikit-learn.

# Download from https://www.anaconda.com/download
# Run the graphical installer
# After installation, verify:
conda --version
# conda 24.x.x

python --version
# Python 3.12.x

# List pre-installed packages (there are hundreds)
conda list | head -20

Anaconda is convenient because everything is pre-installed, but the download is large (~800 MB) and you may not need most of the bundled packages.

Option 3 — Miniconda (Recommended for This Course)

Miniconda is a minimal installer that includes only Python, conda, and their dependencies. It gives you the power of conda for environment and package management without the bloat of Anaconda. You install only what you need.

# Download from https://docs.anaconda.com/miniconda/
# Run the installer
# After installation, verify:
conda --version
python --version
⚠ Course Note: Throughout this course we rely heavily on Miniconda as our primary Python distribution. It provides the conda environment manager which makes it easy to create isolated environments, install scientific packages with compiled dependencies (NumPy, Pandas, etc.), and reproduce setups across different machines. All examples and instructions assume conda is available.

Managing Environments with Conda

The conda command replaces both venv and pip for environment and package management:

# Create a new environment with a specific Python version
conda create --name myproject python=3.12

# Activate the environment
conda activate myproject

# Install packages (conda resolves binary dependencies automatically)
conda install numpy pandas matplotlib jupyter

# You can still use pip inside a conda environment
pip install requests

# List installed packages
conda list

# Export environment to a file (for reproducibility)
conda env export > environment.yml

# Recreate environment from file on another machine
conda env create -f environment.yml

# Deactivate
conda deactivate

# Remove an environment entirely
conda env remove --name myproject

Comparison

python.org Installer

Minimal (~30 MB). Uses pip + venv. Best for lightweight scripting or web development.

Anaconda

Full bundle (~800 MB). 250+ packages pre-installed. Includes GUI (Anaconda Navigator). Best if you want everything ready out of the box.

Miniconda ★

Lightweight (~80 MB). Uses conda for environments & packages. Install only what you need. Recommended for this course.

Key Difference: pip vs. conda

pip installs Python-only packages from PyPI. conda installs packages with compiled C/Fortran dependencies (e.g., NumPy, SciPy) from the Anaconda repository, handling binary compatibility automatically.

1.4 Running Python Code

Python code can be executed in two modes: interactive (REPL) and script-based.

Interactive Mode (REPL)

The Read-Eval-Print Loop is launched by typing python in a terminal. Each expression is evaluated immediately, making it ideal for exploration and debugging.

>>> 2 + 3
5
>>> print("Hello from the REPL")
Hello from the REPL

Script Mode

A Python file (conventionally with a .py extension) is executed by passing it to the interpreter.

# main.py
import sys

print(f"Python version: {sys.version}")
print("Script execution complete.")

1.5 The __name__ Variable and the __main__ Guard

What Is __name__?

Every Python file (module) has a built-in variable called __name__. Python sets its value automatically depending on how the file is being used:

  • ●If the file is run directly (e.g., python myfile.py), Python sets __name__ to the string "__main__".
  • ●If the file is imported by another file (e.g., import myfile), Python sets __name__ to the module's name — in this case "myfile".

You can verify this yourself:

# check_name.py
print(f"__name__ = {__name__}")
# Run directly:
# $ python check_name.py
# __name__ = __main__

# Import from another file:
# >>> import check_name
# __name__ = check_name

The Problem: Unwanted Code Execution on Import

Imagine you write a utility file with useful functions and some code that uses those functions for testing or demonstration. When another file imports your module to reuse the functions, all the top-level code runs too, which is not what you want.

Consider this example with two files:

# math_utils.py  —  A utility module

def add(a, b):
    return a + b

def multiply(a, b):
    return a * b

# Some test code we wrote while developing
print("Testing math_utils...")
print(f"add(2, 3) = {add(2, 3)}")
print(f"multiply(4, 5) = {multiply(4, 5)}")
print("All tests passed!")
# app.py  —  Our main application
import math_utils

result = math_utils.add(10, 20)
print(f"Result: {result}")

When you run python app.py, the output is:

Testing math_utils...
add(2, 3) = 5
multiply(4, 5) = 20
All tests passed!
Result: 30

The test output from math_utils.py leaks into your application! This happens because Python executes every line in a module when it is imported.

The Solution: if __name__ == "__main__"

Wrap any code that should only run when the file is executed directly — not when imported — inside this guard:

# math_utils.py  —  Fixed with __name__ guard

def add(a, b):
    return a + b

def multiply(a, b):
    return a * b

if __name__ == "__main__":
    # This block ONLY runs when you do: python math_utils.py
    # It does NOT run when another file does: import math_utils
    print("Testing math_utils...")
    print(f"add(2, 3) = {add(2, 3)}")
    print(f"multiply(4, 5) = {multiply(4, 5)}")
    print("All tests passed!")

Now when you run python app.py:

Result: 30

Clean! Only your application output. But you can still run python math_utils.py directly to see the test output:

Testing math_utils...
add(2, 3) = 5
multiply(4, 5) = 20
All tests passed!

How It Works

python math_utils.py

__name__ is set to "__main__"
The guard condition is True
Test code runs

import math_utils

__name__ is set to "math_utils"
The guard condition is False
Test code is skipped

Best Practice: Always use the if __name__ == "__main__" guard in any Python file that could be both run directly and imported. This is a fundamental Python pattern you will see in virtually every well-structured project.

1.6 IPython — The Enhanced Interactive Shell

While the default Python REPL is functional, it is minimal. IPython (Interactive Python) is a powerful replacement that adds features every developer will appreciate: syntax highlighting, tab completion, inline help, magic commands, and much more. It is the same kernel that powers Jupyter notebooks.

Installation

IPython is a third-party package installed via pip. It is recommended to install it inside a virtual environment.

# Install IPython
pip install ipython

# Launch it
ipython

Key Features

Once inside the IPython shell you immediately notice several enhancements over the standard REPL:

1. Syntax Highlighting and Smart Indentation

Code is color-highlighted as you type, making it far easier to spot errors and read complex expressions.

In [1]: def factorial(n):
   ...:     if n <= 1:
   ...:         return 1
   ...:     return n * factorial(n - 1)
   ...: 

In [2]: factorial(10)
Out[2]: 3628800

2. Tab Completion

Press Tab after a partial name to auto-complete variables, functions, methods, file paths, and even dictionary keys.

In [3]: import os
In [4]: os.path.<Tab>
# Shows: os.path.join, os.path.exists, os.path.dirname, ...

3. Inline Help with ? and ??

Append ? to any object to see its docstring. Use ?? to see the full source code.

In [5]: len?
# Signature: len(obj, /)
# Docstring: Return the number of items in a container.

In [6]: factorial??
# Shows the full source code of your function

4. Magic Commands

IPython provides special commands prefixed with % (line magic) or %% (cell magic) for common tasks:

# Time a single statement
In [7]: %timeit sum(range(10000))
# 112 µs ± 1.2 µs per loop

# Run an external script
In [8]: %run my_script.py

# List all variables in the namespace
In [9]: %whos
Variable   Type      Data/Info
-------------------------------
factorial  function  <function factorial at 0x...>

# Change directory
In [10]: %cd /path/to/project

# Command history
In [11]: %history -n 1-5

5. Shell Integration

Prefix any line with ! to run it as a system shell command directly from IPython.

In [12]: !pip list | head -5
In [13]: files = !ls *.py
In [14]: print(files)
# ['main.py', 'utils.py', 'test_app.py']

6. Input/Output History

IPython stores every input and output. Access previous results using In[n] and Out[n], or use _ for the last output.

In [15]: 2 ** 10
Out[15]: 1024

In [16]: _ * 2
Out[16]: 2048

In [17]: Out[15]
Out[17]: 1024
IPython is an excellent tool for exploration, quick debugging, and learning. Many experienced Python developers use ipython as their default interactive shell instead of the plain python REPL.

1.7 Virtual Environments and Package Management

A virtual environment is an isolated directory tree that contains a specific Python interpreter and a set of installed packages. This prevents version conflicts between projects.

# Create a virtual environment
python -m venv myenv

# Activate (Windows)
myenv\Scripts\activate

# Activate (Linux / macOS)
source myenv/bin/activate

# Install packages
pip install requests numpy

# Freeze current dependencies
pip freeze > requirements.txt

# Reproduce environment elsewhere
pip install -r requirements.txt
Always work inside a virtual environment to keep your system Python installation clean and to ensure reproducibility of your projects.

1.8 Development Workflow Overview

A typical Python development cycle involves the following steps:

  1. Explore — Use the REPL to test small ideas and examine APIs.
  2. Write — Organize code into modules and packages within a project directory.
  3. Test — Run unit tests with pytest or unittest after each change.
  4. Debug — Use print(), the built-in pdb debugger, or IDE breakpoints.
  5. Ship — Package the project and manage dependencies with pip and pyproject.toml.

This tight feedback loop is one of Python's greatest strengths. Unlike C++ or Java, there is no waiting for a compilation step, which significantly accelerates prototyping and iterative development.
